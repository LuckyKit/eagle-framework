# /eagle:discuss — 需求澄清

> 原子 Skill。只负责一件事：通过多轮对话把需求澄清清楚，输出 DISCUSSION.md。
> 不做规划，不写代码。

---

## 调用方式

```
/eagle:discuss <描述>   ← 用户直接调用
由编排 Skill 调用      ← /eagle:dev / /eagle:fix 内部调用
```

---

## 输入

- 用户描述（需求 / Bug / 重构目标）
- 工作类型提示（由编排 Skill 传入，或自动判断）：
  - `feature` — 新功能
  - `iteration` — 功能迭代
  - `bug` — Bug 修复
  - `refactor` — 重构

---

## 执行步骤

### 1. 判断工作类型

用户已描述时自动判断，无法判断时询问：
```
这是：新功能 / 功能迭代 / Bug 修复 / 重构？
```

---

### 2. 通用澄清

逐步收集，不一次性抛出所有问题：

```
核心目标：做完之后能做到什么？（一句话）
边界：哪些在范围内？哪些明确不做？
涉及端：Go 后端 / Next.js Web / Flutter App？（可多选）
约束：已知技术限制？必须兼容的现有逻辑？
```

---

### 3. Bug 专项澄清（type = bug 时）

```
当前行为：发生了什么？（可粘贴堆栈 / 日志）
期望行为：正确情况下应该如何？
复现条件：如何触发？必现 / 偶发？
影响范围：线上 / 本地？严重程度？
最近变更：有无相关近期改动？（可选）
```

---

### 4. 前端专项澄清（涉及 Next.js 或 Flutter 时）

```
交互流程：用户从哪里进 → 做什么 → 到哪里去？
状态变化：loading / error / empty / success 各什么表现？
输入与校验：需要哪些用户输入？校验规则？
竞品参考：有参考截图吗？可以直接粘贴。
         （粘贴后追问：哪些要参考？哪些不要？）
视觉风格：颜色 / 字体 / 间距有约束吗？整体偏向？
```

---

### 5. 多轮直到清晰

持续追问直到以下条件全部满足：
- 核心目标明确
- 边界清楚
- 涉及端确认
- 前端涉及时：交互流程 + 视觉风格已定

**不限轮数，澄清到位为止。**

---

### 6. 确认摘要

输出理解摘要，等待用户确认：

```
📋 需求理解

类型：{feature / bug / iteration / refactor}
目标：{一句话}
范围：
  - {做的事1}
  - {做的事2}
不做：{明确排除的事}
涉及端：{Go / Next.js / Flutter}

{如有前端}
交互流程：{描述}
视觉参考：{竞品截图要点摘要}

---
还有补充吗？没问题请确认，我去规划方案。
```

用户确认后，写入 DISCUSSION.md，结束。

---

## 输出

文件：`.eagle/tasks/{slug}/DISCUSSION.md`

```markdown
# 需求澄清记录

## 基本信息
- 类型：{feature/bug/iteration/refactor}
- 涉及端：{go/nextjs/flutter}

## 目标
{一句话目标}

## 范围
- {做的事}

## 不做
- {排除的事}

## 约束
{技术约束}

## 前端设计（如有）
### 交互流程
{描述}

### 视觉风格
{描述}

### 竞品参考要点
{从截图中提炼的关键点}
```

---

## 硬性约束

1. **不做规划** — 只澄清，不生成 Phase / Wave
2. **不写代码** — 不给出实现建议
3. **前端必须澄清交互** — 只要涉及 Next.js / Flutter，交互流程和视觉风格不能跳过
4. **竞品截图必须追问** — 粘贴截图后必须问"哪些参考，哪些不要"
5. **用户确认后才输出 DISCUSSION.md** — 不要在确认前写文件
